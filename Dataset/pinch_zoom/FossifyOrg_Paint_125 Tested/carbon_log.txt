Steps to reproduce the bug
sometimes paint goes to "draw to wrong place" bug mode. This happens typically after first drawing something with pencil and then zooming out with 2 fingers. Also I now noticed that the bug appears sometimes when I turn the device from portrait to landscape or back (auto rotate android UI is enabled).


When bug is present if I touch pixel 300,300 and pull it to pixel 600,600, the actual line on the screen could appear to pixels 600,100 - 900,400.


The length of error vector varies, sometimes it could be 25% of the length of display and sometimes only 2-5%


Getting rid of the bug I have to go to settings - apps - Fossify Paint - force stop.


Device is Oneplus Pad Go (Mediatek G99 soc, 8GB RAM, 2480 x 1720 resolution (1.4 aspect ratio), 11 inch screen).


I did just 5min test with my phone (galaxy S21fe, 2300ish x 1080 res) and in that time I was unable to reproduce it.


Worth noting is that this tablet has unusual aspect ratio and resolution for androids (typical android tablets are 1.6 aspect ratio and phones are over 2). Nevertheless IMO this is best aspect ratio and e.g. iPad aspect ratios are at same ballpark (1.33 - 1.5).


With tablet it is much more common to rotate the device than with phones (IMO) so program should take this into account.


The bug does not appear every time, I am not sure what factors cause it. But if you take a tablet, start drawing, doing lot of 2 finger pinch zooms in and out and drawing in different zooming factors, sometimes panning (moving 2 fingers to any directions), and then rotating the device so that auto-rotate happens you have high change of running into this bug.


I suspect that the bug is on the algorithm which translates touch coordinate to drawn pixel coordinate.


Expected behavior
pen location remains in touch location


Actual behavior
pen location is different than touch location

Why ReBL fails on this bug
The old ReBL has no multi-touch capability at all. Its handle_command.py only supports single-finger actions: click, long_click, swipe, scroll, set_text. When it encounters a bug report requiring pinch zoom, it has no action to execute — it either skips the step or tries to approximate it with two sequential single-finger swipes, which the app's ScaleGestureDetector never recognises as a zoom gesture. The canvas zoom state never changes, so the coordinate mapping bug is never triggered.
Why CARBON can reproduce it
Three capabilities work together:
1. Real multi-touch via UiAutomator2 jsonrpc — device.jsonrpc.pinchIn/pinchOut calls the UiAutomator2 server's native UiObject2.pinchIn/pinchOut, which uses Android's GestureController.performGesture with two simultaneous PointerGesture objects. These are injected as real MotionEvent.ACTION_POINTER_DOWN events with two active pointer IDs, which is exactly what ScaleGestureDetector requires to fire onScaleBegin/onScale/onScaleEnd. The canvas zoom state actually changes.

2. Visual context via UIAutomator Viewer — The annotated screenshot shows the LLM the actual canvas state after each zoom. The LLM can see the drawn line appearing smaller (zoom out) or larger (zoom in) and confirm the zoom worked before proceeding to the next step. Old ReBL is text-only — it cannot observe visual state changes on a custom-drawn canvas that has no accessibility tree representation.

3. Coordinate-based drawing via swipe_region — The bug manifests as a coordinate offset between touch input and canvas rendering. Old ReBL's click and swipe actions target UI elements by text/resource-id, not by raw coordinates. It cannot draw at a specific (300, 300) to (600, 600) path and then observe whether the resulting line appears at a different location. New ReBL's swipe_region with explicit coordinates, combined with the visual screenshot, lets the LLM compare intended vs actual drawing position.

The core argument for the paper
This bug class — coordinate mapping failures in custom-drawn views after gesture state changes — is fundamentally invisible to text-only automation. The accessibility tree reports View bounds=[0,374][1080,1889] regardless of zoom level. There is no text, no content description, no resource-id that changes when the canvas zooms. The only observable evidence is pixel-level: the drawn line appears at a different position than the touch coordinates. Old ReBL has no mechanism to observe this. New ReBL's multimodal pipeline (screenshot + color analysis + coordinate-based actions) is the minimum capability required to detect and reproduce this entire class of visual rendering bugs.
