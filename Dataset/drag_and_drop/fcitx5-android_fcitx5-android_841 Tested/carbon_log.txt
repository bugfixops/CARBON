CARBON already has drag_and_drop — it's in handle_command.py using device.drag(x1, y1, x2, y2, duration). The problem we hit was specifically with ItemTouchHelper reorder lists where device.drag() triggers multi-select instead of reorder mode.


For general drag-and-drop (moving widgets on home screen, dragging files, dragging sliders), it works fine. The limitation is only with RecyclerView items that use Android's ItemTouchHelper pattern — those expect a specific hold-then-move touch sequence that device.drag() doesn't produce correctly.


There's no way to fix this within uiautomator2's API. The device.drag() function does a smooth linear interpolation from start to end, but ItemTouchHelper needs a distinct long-press phase (to enter drag mode) followed by a separate move phase. These are fundamentally different touch patterns and uiautomator2 doesn't expose the low-level control needed to separate them.


So the situation is:


General drag-and-drop (widgets, sliders, files): works
ItemTouchHelper reorder (list item reordering): doesn't work — this is a uiautomator2 limitation, not something we can fix in CARBON
